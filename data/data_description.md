# ExperimentMind Pay 数据字段说明

## 一、数据表定位

当前项目使用的是：

> **用户级支付实验分析表**

它并不是逐笔支付订单表，也不是原始支付事件日志。

数据粒度规定为：

```text
一行 = 一个用户在整个实验观察期内的汇总结果
```

例如，某个用户在实验期间可能多次打开支付页面、多次尝试支付或完成多笔订单，但在当前数据表中，该用户最终只保留一行汇总记录。

系统默认实验按照用户进行随机分组，因此：

* 每个用户只能属于一个实验组；
* 每个用户只能出现一次；
* 指标计算和统计检验以用户为分析单位。

---

## 二、字段总览

| 字段名                  | 中文名称      |   数据类型 | 是否必填 | 示例       |
| -------------------- | --------- | -----: | ---: | -------- |
| `user_id`            | 用户唯一标识    | 字符串或整数 |    是 | `10001`  |
| `group`              | 实验分组      |    字符串 |    是 | `A`      |
| `exposed`            | 是否曝光实验页面  |     整数 |    是 | `1`      |
| `attempted`          | 是否发起支付    |     整数 |    是 | `1`      |
| `paid`               | 是否支付成功    |     整数 |    是 | `1`      |
| `payment_amount`     | 实验期内支付总金额 |    浮点数 |    是 | `128.50` |
| `refunded`           | 是否发生退款    |     整数 |    是 | `0`      |
| `refund_amount`      | 实验期内退款总金额 |    浮点数 |    是 | `0.00`   |
| `payment_latency_ms` | 支付流程耗时    | 整数或浮点数 |    是 | `1350`   |
| `device`             | 用户设备类型    |    字符串 |    是 | `ios`    |
| `new_user`           | 是否为新用户    |     整数 |    是 | `1`      |

## 三、字段详细说明

### 1. `user_id`

**中文名称：** 用户唯一标识

**含义：**

用于唯一识别实验中的用户。

当前数据表以用户为分析单位，因此每个 `user_id` 原则上只能出现一次。

**推荐类型：**

```text
字符串
```

即使用户编号当前全部由数字组成，也建议在真实项目中按字符串处理，因为用户编号可能：

* 包含字母；
* 包含前导零；
* 超过普通整数范围；
* 来自不同业务系统。

**示例：**

```text
10001
U202607130001
00001235
```

**数据约束：**

* 不允许为空；
* 不允许重复；
* 同一用户不能同时进入 A、B 两组；
* 不建议使用用户姓名、手机号等直接身份信息。

---

### 2. `group`

**中文名称：** 实验分组

**含义：**

表示用户被分配到哪个实验版本。

当前项目约定：

```text
A = 对照组
B = 实验组
```

**示例：**

```text
A
B
```

**数据约束：**

* 只能包含 `A` 或 `B`；
* A、B 两组必须同时存在；
* 不允许为空；
* 同一个用户只能属于一个组；
* 分组原则上应在实验开始前确定，不能根据支付结果事后调整。

---

### 3. `exposed`

**中文名称：** 是否曝光实验页面

**含义：**

表示用户在实验观察期内是否至少看到过一次实验页面、支付页面或实验策略。

**取值：**

```text
1 = 至少曝光过一次
0 = 没有曝光
```

**示例：**

```text
1
```

**数据约束：**

* 只能为 `0` 或 `1`；
* 不允许为空；
* 若 `attempted = 1`，则原则上必须满足 `exposed = 1`；
* 若用户没有曝光，则通常不应发生由该实验页面触发的支付行为。

**说明：**

当前字段表示“是否至少曝光过一次”，并不表示曝光次数。

如果未来需要分析曝光频次，可增加：

```text
exposure_count
```

---

### 4. `attempted`

**中文名称：** 是否发起支付

**含义：**

表示用户在实验观察期内是否至少发起过一次支付尝试。

**取值：**

```text
1 = 至少发起过一次支付
0 = 没有发起支付
```

**数据约束：**

* 只能为 `0` 或 `1`；
* 不允许为空；
* 应满足：

```text
attempted <= exposed
```

也就是说，没有看到支付页面的用户，原则上不应发起由该页面产生的支付。

**说明：**

当前字段只表示是否尝试过，不记录尝试次数。

未来可增加：

```text
payment_attempt_count
```

用于统计用户在实验期间总共尝试了多少次支付。

---

### 5. `paid`

**中文名称：** 是否支付成功

**含义：**

表示用户在实验观察期内是否至少完成过一次支付。

**取值：**

```text
1 = 至少成功支付过一次
0 = 没有成功支付
```

**数据约束：**

* 只能为 `0` 或 `1`；
* 不允许为空；
* 应满足：

```text
paid <= attempted
```

即没有发起支付的用户，不应被标记为支付成功。

**说明：**

当前字段表示“是否至少成功过一次”，不表示成功订单数量。

未来可增加：

```text
successful_payment_count
```

用于表示用户在实验期间的支付成功订单数。

---

### 6. `payment_amount`

**中文名称：** 实验期内支付总金额

**含义：**

表示该用户在实验观察期内所有支付成功订单的金额总和。

**单位：**

默认使用人民币元。

例如：

```text
128.50
```

表示用户在实验期间累计支付了 128.50 元。

**数据约束：**

* 不允许为空；
* 必须大于等于 `0`；
* 若 `paid = 0`，原则上应满足：

```text
payment_amount = 0
```

* 若 `paid = 1`，通常应满足：

```text
payment_amount > 0
```

但在零元订单、全额抵扣等特殊业务中，可能出现支付成功但金额为 0，需要根据业务规则判断。

**说明：**

该字段不是单笔订单金额，而是用户实验期内的累计支付金额。

---

### 7. `refunded`

**中文名称：** 是否发生退款

**含义：**

表示用户在实验观察期或指定退款观察期内，是否至少发生过一次退款。

**取值：**

```text
1 = 至少发生过一次退款
0 = 没有退款
```

**数据约束：**

* 只能为 `0` 或 `1`；
* 不允许为空；
* 一般应满足：

```text
refunded <= paid
```

即用户没有支付成功时，原则上不应发生退款。

**说明：**

退款可能发生在实验结束后，因此真实项目中应明确退款观察窗口，例如：

```text
支付后 7 天内是否退款
支付后 30 天内是否退款
```

---

### 8. `refund_amount`

**中文名称：** 实验期内退款总金额

**含义：**

表示用户在规定退款观察期内发生的累计退款金额。

**单位：**

默认使用人民币元。

**数据约束：**

* 不允许为空；
* 必须大于等于 `0`；
* 若 `refunded = 0`，原则上应满足：

```text
refund_amount = 0
```

* 一般应满足：

```text
refund_amount <= payment_amount
```

如果退款观察窗口和支付观察窗口不同，或者存在历史订单退款，则需要单独说明。

**示例：**

```text
0
58.50
128.50
```

---

### 9. `payment_latency_ms`

**中文名称：** 支付流程耗时

**含义：**

表示用户从发起支付到支付结果返回所经历的时间。

**单位：**

毫秒。

例如：

```text
1350
```

表示支付耗时 1350 毫秒，即 1.35 秒。

**当前推荐口径：**

为了保证一行一个用户，建议第一版明确为：

> 用户首次支付尝试的支付结果耗时。

或者：

> 用户首次成功支付的耗时。

项目必须在 README 中固定其中一种口径，不能混用。

**数据约束：**

* 必须大于等于 `0`；
* 若 `attempted = 0`，建议设置为 `0` 或空值；
* 不能出现负数；
* 超过合理范围时应产生警告；
* 不同记录必须使用统一单位。

**说明：**

当前样例数据中使用 `0` 表示用户没有发起支付。

在更严格的数据设计中，也可以使用空值表示“不适用”。

---

### 10. `device`

**中文名称：** 用户设备类型

**含义：**

表示用户参与实验或发起支付时使用的主要设备或客户端类型。

**当前示例取值：**

```text
ios
android
```

未来可以扩展为：

```text
web
h5
mini_program
other
```

**数据约束：**

* 不允许为空；
* 推荐统一使用小写；
* 不应同时出现含义相同但格式不同的值，例如：

```text
iOS
IOS
ios
```

应统一为：

```text
ios
```

**用途：**

用于设备分群分析，例如比较：

* iOS 用户实验效果；
* Android 用户实验效果；
* 不同客户端是否存在明显差异。

---

### 11. `new_user`

**中文名称：** 是否为新用户

**含义：**

表示用户在实验开始时是否被定义为新用户。

**取值：**

```text
1 = 新用户
0 = 老用户
```

**数据约束：**

* 只能为 `0` 或 `1`；
* 不允许为空；
* 新用户定义必须在整个实验中保持一致。

**新用户定义示例：**

可以按以下任一规则定义：

```text
注册时间不超过 7 天
实验前从未完成支付
实验前 30 天没有支付行为
```

实际项目必须选择一种明确规则，不能临时改变。

**用途：**

用于判断实验效果是否主要来自：

* 新用户；
* 老用户；
* 某一类用户群体。

## 四、字段之间的业务逻辑关系

支付漏斗通常满足：

```text
exposed >= attempted >= paid >= refunded
```

也可以写为：

```text
attempted <= exposed
paid <= attempted
refunded <= paid
```

典型合法记录：

```csv
user_id,group,exposed,attempted,paid,payment_amount,refunded,refund_amount
10001,A,1,1,1,128.50,0,0
```

典型非法记录：

```csv
10002,A,0,1,1,99.00,0,0
```

原因：

```text
用户没有曝光，却发起并完成了支付。
```

另一个非法示例：

```csv
10003,B,1,1,0,88.00,0,0
```

原因：

```text
用户未支付成功，但支付金额大于 0。
```

## 五、当前数据表可以计算的原子指标

原子指标是直接从字段中计数、去重或求和得到的指标。

### 用户数

```text
COUNT DISTINCT(user_id)
```

### 曝光用户数

```text
SUM(exposed)
```

### 支付尝试用户数

```text
SUM(attempted)
```

### 支付成功用户数

```text
SUM(paid)
```

### 支付金额总和

```text
SUM(payment_amount)
```

### 退款用户数

```text
SUM(refunded)
```

### 退款金额总和

```text
SUM(refund_amount)
```

---

## 六、可以进一步计算的衍生指标

### 支付尝试率

```text
支付尝试用户数 / 曝光用户数
```

### 支付成功率

```text
支付成功用户数 / 曝光用户数
```

### 尝试后支付成功率

```text
支付成功用户数 / 支付尝试用户数
```

### 每曝光用户收入

```text
支付金额总和 / 曝光用户数
```

### 支付用户平均金额

```text
支付金额总和 / 支付成功用户数
```

### 退款用户率

```text
退款用户数 / 支付成功用户数
```

### 每用户净收入

```text
(支付金额总和 - 退款金额总和) / 曝光用户数
```

## 七、当前数据格式的限制

当前用户级汇总表无法完整表示以下信息：

* 用户进行了多少次支付尝试；
* 用户完成了多少笔订单；
* 每一笔订单的具体金额；
* 每次支付发生的时间；
* 支付失败原因；
* 一笔订单是否发生多次退款；
* 用户多次支付的耗时分布；
* 支付状态的完整变化过程。

因此，该数据表适合用于第一阶段的实验指标分析，但不等同于完整支付交易数据。

后续版本可以增加：

* 实验分组表；
* 原始支付事件表；
* 订单表；
* 退款表；
* 用户级聚合模块。

完整数据流程将扩展为：

```text
实验分组数据
+ 支付事件数据
+ 订单与退款数据
→ 清洗和关联
→ 用户级聚合
→ 实验指标计算
→ 统计检验
→ 上线决策
```

## 九、数据使用原则

系统在进行任何指标计算前，应依次检查：

```text
文件是否存在
→ 数据是否为空
→ 必要字段是否齐全
→ 必要值是否缺失
→ 用户是否重复
→ 实验分组是否合法
→ 二元字段是否合法
→ 支付漏斗是否自洽
→ 金额字段是否合法
→ 支付耗时是否合理
```

只有通过数据质量验证的数据，才允许进入后续指标计算和统计检验流程。
